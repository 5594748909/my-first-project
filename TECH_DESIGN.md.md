# 一、文档概述

## 1.1 设计目标

本技术设计文档基于**海龟汤游戏PRD V1.0**制定，围绕轻量化、高可用、易扩展、低成本四大核心原则，完成全链路技术方案设计，保障产品功能落地、性能达标、合规安全，同时支撑后续玩法迭代与用户规模扩容。

## 1.2 设计原则

- **轻量化优先**：适配小程序/H5优先部署，减少资源占用，降低加载时延

- **高可用性**：核心服务无单点故障，断线重连、数据兜底机制完善

- **易扩展性**：模块化拆分设计，支持玩法新增、题库扩容、并发升级

- **安全合规**：数据加密、内容审核、未成年人保护全流程合规

- **低成本运维**：选用云原生轻量化组件，简化部署与监控流程

## 1.3 技术范围

覆盖前端（小程序/H5）、后端服务、数据存储、消息推送、实时通讯、内容审核、运维监控全技术栈，不含第三方支付、深度IP联动等扩展功能（预留接口）。

# 二、整体技术架构

## 2.1 总体架构分层

采用**前后端分离 + 微服务轻量化（单体优先，微服务拆分预留）**架构，分为接入层、网关层、业务服务层、数据层、第三方依赖层，整体架构简洁高效，适配初创期小规模并发，后期平滑扩容。

### 2.1.1 架构分层详解

- **接入层**：微信小程序、H5移动端页面，负责UI渲染、交互逻辑、本地缓存

- **网关层**：请求路由、权限校验、限流熔断、接口转发、日志采集

- **业务服务层**：核心业务模块拆分，包含用户服务、游戏服务、内容服务、社交服务、运营服务

- **数据层**：关系型数据库、缓存数据库、对象存储、日志存储

- **第三方依赖层**：微信开放平台、内容安全审核、短信服务、实时通讯SDK

## 2.2 核心技术选型

|技术模块|选型方案|选型理由|
|---|---|---|
|前端框架|微信小程序原生、uni-app（H5适配）|轻量化、跨端兼容、开发效率高，适配小程序优先策略|
|后端语言|Java（SpringBoot 2.7.x）|生态成熟、稳定性强、易运维，适配中小团队研发|
|关系型数据库|MySQL 8.0|开源免费、事务支持强，适配用户、题库、游戏记录等结构化数据|
|缓存数据库|Redis 6.2|高性能KV存储，用于会话缓存、排行榜、限流、热点数据缓存|
|实时通讯|WebSocket + 第三方IM SDK（备选）|满足多人联机实时提问、聊天、状态同步|
|内容审核|阿里云/腾讯云内容安全API|文本智能审核，过滤违规内容，降低人工审核压力|
|对象存储|腾讯云COS/阿里云OSS|存储用户头像、汤面配图、日志文件等静态资源|
|部署方案|Docker容器化 + 云服务器ECS|环境统一、部署快捷、弹性扩容方便|
## 2.3 核心流程时序

### 2.3.1 单人游戏核心时序

1. 前端请求题库列表 → 网关校验权限 → 游戏服务查询MySQL题库数据 → Redis缓存热点题目

2. 用户进入游戏，前端展示汤面，用户提交提问 → 后端语义判定 → 返回是/否/无关结果

3. 用户推理完成 → 提交答案校验 → 记录游戏数据 → 发放奖励、更新成就

### 2.3.2 多人联机核心时序

1. 房主创建房间 → 后端生成房间ID → 存入Redis缓存房间信息

2. 玩家加入房间 → WebSocket建立长连接 → 同步房间状态、成员信息

3. 玩家提问 → 消息广播至房间全员 → 汤主回复 → 实时同步回复结果

4. 猜出汤底 → 结算奖励 → 销毁房间缓存/保留重开状态

# 三、前端技术设计

## 3.1 前端架构设计

采用**MVC分层模式**，拆分页面层、逻辑层、数据请求层、工具层，降低耦合度，提升复用性。

- **页面层**：首页、题库页、单人游戏页、多人房间页、个人中心、UGC投稿页

- **逻辑层**：游戏交互逻辑、房间管理逻辑、提问判定逻辑、成就计算逻辑

- **数据层**：接口请求封装、本地缓存管理、状态管理

- **工具层**：表单校验、文本过滤、时间格式化、异常处理

## 3.2 核心模块实现

### 3.2.1 单人游戏模块

- **提问交互**：输入框字数限制（≤20字）、敏感词前端预过滤、防抖提交

- **推理记录**：本地缓存历史问答，支持下拉刷新、清空记录，断线不丢失

- **道具交互**：提示、跳过、查看汤底按钮状态控制，道具消耗校验

### 3.2.2 多人联机模块

- **WebSocket连接**：心跳检测、断线重连、超时自动断开，重连后恢复房间状态

- **房间管理**：房间列表渲染、成员状态同步、禁言/踢人前端权限控制

- **消息交互**：提问消息、回复消息、系统通知区分样式，滚动定位最新消息

### 3.2.3 UGC投稿模块

- 汤面/汤底输入校验、分类选择、标签绑定，敏感词预校验

- 提交状态反馈（审核中、已通过、已驳回），驳回原因展示

## 3.3 性能优化方案

- 静态资源压缩、图片懒加载、分包加载（小程序），首屏加载≤2s

- 接口数据缓存、重复请求防抖、接口合并，减少网络请求次数

- 本地缓存用户进度、题库列表，降低重复请求开销

# 四、后端技术设计

## 4.1 后端服务拆分

初创期采用**单体应用拆分模块**，后期可平滑拆分为微服务，核心模块如下：

- **用户中心服务**：注册登录、账号绑定、个人信息、权限管理、青少年模式

- **游戏核心服务**：题库管理、单人游戏、多人房间、提问判定、结算奖励

- **内容管理服务**：UGC投稿、内容审核、素材管理、违规处理

- **成长体系服务**：等级计算、成就解锁、排行榜、经验/金币管理

- **运营支撑服务**：公告管理、活动配置、福利发放、数据统计

- **网关公共服务**：全局拦截、限流熔断、日志采集、异常处理

## 4.2 核心接口设计

采用RESTful接口规范，统一返回格式：`{code:200/500, msg:'提示信息', data:{}}`，核心接口如下：

|接口模块|接口名称|请求方式|核心参数|
|---|---|---|---|
|用户模块|微信授权登录|POST|code、encryptedData、iv|
|题库模块|获取题库列表|GET|分类、难度、页码、页数|
|单人游戏|提交提问|POST|题目ID、提问内容|
|多人房间|创建房间|POST|题目ID、房间密码、人数上限|
|UGC模块|投稿汤面|POST|汤面、汤底、分类、难度|
## 4.3 核心业务逻辑

### 4.3.1 提问判定逻辑

针对单人模式，采用**关键词匹配 + 语义规则**判定：

1. 后台配置每道题的关键词库（肯定关键词、否定关键词、无关关键词）

2. 用户提问文本分词，匹配关键词库

3. 匹配肯定词→返回“是”，匹配否定词→返回“否”，均不匹配→返回“无关紧要”

### 4.3.2 多人房间管理

- 房间信息存入Redis，设置过期时间（空闲30分钟自动销毁）

- 房主掉线→自动转让房主给最早加入成员，房间不解散

- 房间人数上限校验、重复加入校验、禁言/踢人权限校验

### 4.3.3 内容审核流程

1. 用户提交UGC内容→调用第三方文本审核API

2. 审核通过→标记待人工复核；审核违规→直接驳回并反馈原因

3. 人工复核通过→上架公共题库；复核不通过→驳回并记录违规次数

## 4.4 缓存设计

- **热点数据**：精选题库、首页公告、排行榜数据，缓存时长1小时

- **临时数据**：多人房间信息、用户会话、游戏进度，缓存时长30分钟-2小时

- **限流数据**：接口请求次数、提问频率，采用Redis计数器实现限流

# 六、非功能技术设计

## 6.1 性能指标

- 接口响应时间：常规接口≤200ms，游戏核心接口≤500ms

- 并发支持：单机支持1000+用户同时在线，200+房间并发

- Redis命中率：≥90%，数据库查询压力降低80%

## 6.2 安全设计

- **接口安全**：Token鉴权、请求签名、防重放攻击、SQL注入过滤

- **数据安全**：用户手机号脱敏、密码加密存储、数据备份每日全量备份

- **防作弊**：提问频率限制、答案提交校验、异常行为监控

## 6.3 高可用设计

- 服务熔断：接口超时/异常自动熔断，返回兜底数据

- 断线重连：WebSocket自动重连，游戏进度不丢失

- 数据兜底：MySQL主从备份，Redis持久化配置

## 6.4 监控与运维

- 服务监控：接口耗时、错误率、在线人数、Redis内存监控

- 日志管理：全局日志采集、异常日志告警、操作日志留存

- 部署方案：Docker容器化，一键部署、弹性扩容

# 七、风险与技术应对

|技术风险|风险描述|应对方案|
|---|---|---|
|实时通讯延迟|多人房间WebSocket卡顿、消息延迟|优化心跳间隔、接入就近节点、备用IMSDK兜底|

|并发压力过大|活动期用户暴涨，服务宕机|限流兜底、Redis集群扩容、服务水平扩容|
|内容审核漏判|违规内容绕过审核上架|多层审核机制、用户举报入口、定时巡检|
